Requirements & Architecture

2026-07-07 · decided via deep research + requirements grilling · research report: research/apple-watch-dashboard-research.md

A single-user, self-hosted web dashboard for Apple Watch health data: wellness trends plus a full quantified-self archive. Apple provides no cloud API, so data is pushed from the iPhone by the Health Auto Export app into our own receiver.

Locked decisions (grilling, 2026-07-07)
Purpose: wellness trends + quantified-self archive. Ingestion: Health Auto Export auto-sync + manual file import (export.zip / HAE JSON upload in UI, repeatable). Hosting: Docker on a home machine, reachable via Tailscale only (no public exposure). Stack: FastAPI + SQLite + React (Recharts). Watch: Series 8+ class → full metric set incl. wrist temperature.
Superseded in part
Charts: Recharts dropped in favour of hand-rolled SVG components ported from the mockup — see page 3 · Build Plan v1.

Architecture

1
Apple Watch → iPhone HealthKit Apple
Watch syncs samples to the iPhone Health store automatically.
2
Health Auto Export app $24.99 lifetime + manual import always available
Two routes into the receiver, same idempotent ingest: (a) HAE scheduled automation POSTs JSON over Tailscale; (b) manual upload of Health app export.zip (or HAE JSON/CSV file) through the dashboard UI — used for initial backfill, repairing sync gaps, or running entirely without HAE. Streaming XML parser handles multi-hundred-MB exports; idempotent upserts make re-imports safe.
3
FastAPI receiver + SQLite Docker
/ingest endpoint (API-key header, idempotent upserts keyed on metric+timestamp+source). Score computation (sleep score, readiness) as query-time or nightly jobs in Python. SQLite file = trivial backup.
4
React + Recharts dashboard Tailscale-only
Served by the same container. No login page — Tailscale is the auth perimeter (single user). Viewable from any of Wannita's devices on the tailnet.

Defaults taken without asking (revisit if they chafe)

  • No dashboard login — Tailscale is the perimeter; ingest still requires an API-key header.
  • Store raw samples, aggregate at query time — keeps archive lossless; HAE set to fine-grained aggregation for HR, defaults elsewhere.
  • SQLite over InfluxDB/Postgres — single user, single writer; millions of rows are fine.
  • Score formulas: readiness = weighted z-scores of overnight HRV, RHR, sleep duration vs 60-day baseline (Training Today / Athlytic pattern); sleep score = duration + stage mix + consistency (Apple's published inputs). Tunable constants, kept in one module.
Risks
(1) HAE automations silently stall under iOS background restrictions — dashboard should show a "last sync" staleness indicator prominently. (2) HAE is closed-source indie software — payload spec is open and receivers are swappable, so migration cost is low. (3) Overnight data arrives in a morning burst after first unlock — score computation must tolerate late-arriving samples.

Open items (for the build plan, page 2)

  • Exact HAE automation config (interval, aggregation per metric group, batching).
