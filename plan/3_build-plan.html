Build Plan v1

2026-07-07 · implements page 1 architecture + page 2 UI spec

Seven milestones, each independently verifiable offline with fixtures before anything touches the phone. Wannita runs the two live steps herself (HAE setup, real export import).

Decisions made here
(1) Frontend charts: port the mockup's hand-rolled SVG builders into React components — no Recharts. The mockup code already implements the band/gap/tooltip specs exactly; a chart lib would fight them. Supersedes the "React (Recharts)" note on page 1. (2) No scores table: readiness/sleep scores computed at query time — late-arriving overnight samples make cached scores stale anyway; 90 days of rows is trivial to recompute. (3) Own backfill parser (~200 lines, iterparse) — healthkit-to-sqlite is frozen since 2021.

SQLite schema

samples(metric TEXT, ts TEXT, value REAL, unit TEXT, source TEXT DEFAULT '',
        PRIMARY KEY (metric, ts, source)) WITHOUT ROWID;   -- idempotent upsert target
sleep_nights(date TEXT PRIMARY KEY, start_ts TEXT, end_ts TEXT,
        deep_h REAL, core_h REAL, rem_h REAL, awake_h REAL, source TEXT);
workouts(id TEXT PRIMARY KEY, type TEXT, start_ts TEXT, end_ts TEXT,
        duration_s REAL, energy_kcal REAL, distance_km REAL, avg_hr REAL,
        raw_json TEXT);                                    -- stored in v1, UI in v2
ingest_log(ts TEXT, kind TEXT, n_rows INTEGER, detail TEXT);  -- feeds last-sync pill

One wide samples table (metric name = HAE metric key, normalized units at ingest); per-night sleep and workouts get their own shape. Re-import and overlapping HAE posts are safe by construction: INSERT OR REPLACE on the natural key.

API

RoutePurpose
POST /api/ingestHAE automation target. Requires X-API-Key header. Accepts HAE JSON (batched); returns rows written.
POST /api/importManual upload: export.zip or HAE JSON/CSV file. Streams zip → iterparse → upserts in a background task; returns job id.
GET /api/import/{id}Import progress (records parsed / written) for the UI progress bar.
GET /api/summary?days=NEverything the main view needs in one call: scores + deltas, headlines, sparkline arrays, last-sync.
GET /api/series/{metric}?days=NValues + rolling mean ±1.5σ band (computed in queries.py).
GET /api/sleep?days=NPer-night stage rows.

Score formulas (v0 — tunable constants in scores.py)

  • Readiness (0–100) = 50 + 25·z(HRV overnight) − 15·z(RHR) + 10·z(sleep duration), z vs trailing 60-day mean/σ, clamped; components returned so the UI can render the explanation sentence ("HRV below your normal range").
  • Sleep score (0–100) = 50·(duration/8h capped) + 30·(deep+REM fraction vs 0.35 target) + 20·(bedtime within ±45 min of 14-day median). Same structure Apple publishes as inputs.
  • Both recomputed on read; formulas versioned in one dict so retuning never touches query code.

Risks watched during build
HAE payload field names must be confirmed against a real device post at milestone 6 — fixtures derive from docs, not ground truth; keep ingest.py tolerant (unknown metrics → stored raw, never dropped). Late overnight data: scores computed on read handles it. OneDrive path: SQLite volume lives in the Docker volume, NOT under OneDrive (file-locking corruption risk).

Explicitly out of v1

Workout/GPS pages, noise + mindfulness UI (v2 — data already stored), auth UI (Tailscale is the perimeter), multi-user, HTTPS (add via tailscale serve later if wanted).
